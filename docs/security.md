# 安全模型与边界

## 保护目标

- 打卡用户身份来自签名 JWT，而不是客户端提交的用户 ID。
- 同一用户的状态流转在数据库中串行，不能重复上班或越过上班直接下班。
- 网络重试不能生成重复记录。
- 考勤时间和工时只由服务器时间决定。
- 未登记设备、明显模拟器信号和异常触控请求在落库前拒绝。
- 审计记录保留客户端时间、服务端时间、IP、设备信息和触控指标。

## 已实现控制

JWT 使用 HS256、短期 `exp` 和至少 32 字节密钥。生产环境不提供内置登录或用户注册，正式身份生命周期应由外部 IdP 负责。开发令牌路由默认不注册。

设备指纹由安全存储中的随机安装 ID、包名和有限设备属性做 SHA-256 得到。服务端只存哈希，不接收随机安装 ID。卸载重装会生成新指纹并要求重新登记。

设备登记令牌通过 HTTPS header 发送且不在客户端持久化。当前后端配置使用共享登记令牌，适合首版和受控团队；生产升级应改为用户绑定、一次性、短时有效的登记凭据。

打卡事务锁定 `attendance_states` 用户行。`attendance_records(user_id, request_id)` 唯一约束提供最终幂等保证。Redis 缓存不参与正确性判定。

Redis Lua 脚本原子执行 IP 计数与首次过期。`REDIS_FAIL_CLOSED` 决定风险服务不可用时拒绝还是降级放行。只有在明确配置受信任反向代理后才读取 `X-Forwarded-For`。

## 不能单独证明身份的信号

`local_auth` 是重要的误操作和普通代打卡屏障，但服务端无法验证客户端是否真的弹出了系统生物识别。被 root、注入或重打包的客户端可以绕过本地逻辑。高风险部署应增加 Play Integrity、App Attest 或硬件密钥签名挑战。

`is_physical_device`、系统版本和设备型号由客户端上报，攻击者可以伪造。它们适合风险评分，不能替代密钥证明。

按压时长、距离和采样数能拦截低成本脚本，但自动化工具也可以模拟真实分布。服务端应持续分析群体分布并设置复核而非仅靠固定阈值永久封禁。

客户端时间差检查只能发现明显错误或过期请求。攻击者可以提交接近服务器的时间，因此实际记录仍只使用 `Utc::now()`。

## IP 与位置

系统不请求 GPS。后端记录网络 IP，但没有声称仅凭 IP 完成北京到纽约的异常跳变判断。可靠实现需要定期更新的 GeoIP/ASN 数据库、VPN/代理策略、置信度阈值、IPv6 与移动网络处理，以及误判申诉通道。

## 隐私

设备和 IP 数据属于安全审计数据，应设置保留期限、访问控制和删除流程。不要采集不可用或高噪声的“电池健康度”作为稳定身份，也不要将原始系统标识符拼接进分析日志。

屏幕尺寸只作为本次审计元数据，不参与稳定指纹，避免旋转、分屏或显示缩放导致设备身份变化。

## 上线清单

- 全链路 TLS，反向代理移除用户提供的 forwarding headers。
- 关闭 `ALLOW_DEV_AUTH`，替换所有示例密钥并建立轮换机制。
- JWT 密钥放入 Secret Manager，不写入镜像、仓库或 `--dart-define`。
- 将共享设备登记令牌替换为短期一次性凭据。
- 启用数据库备份、迁移演练、结构化审计日志与告警。
- 定义 Redis fail-open/fail-closed 的业务策略并演练故障。
- 增加设备完整性证明和签名 challenge，检测重放。
- 为用户保存 IANA 时区，明确跨午夜、夏令时和跨时区工时归属。
- 对数据保留、管理员访问和员工知情取得合规确认。
